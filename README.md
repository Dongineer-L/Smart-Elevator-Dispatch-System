🏢 智能电梯调度模拟系统 (Smart Elevator Dispatch System)
一个基于原生 JavaScript 和 Web Workers 实现的高仿真电梯调度系统。该项目模拟了现代写字楼中多部电梯的协同运行，包含拟物化的 UI 界面、真实的物理运行逻辑以及高效的调度算法。
✨ 核心特性
● ⚡️ 多线程架构: 每一部电梯都在独立的 Web Worker 线程中运行，互不阻塞，模拟真实的独立控制器。
● 🧠 智能调度算法: 采用 LOOK 算法（扫描算法）作为单梯运行策略，结合主线程的“代价评估函数”进行多梯任务分配，实现最优响应。
● 🎨 拟物化 UI 设计:
  ○ 纯 CSS 实现的 3D 质感电梯井与轿厢。
  ○ 动态开关门动画与内部灯光效果。
  ○ 实时高亮的楼层按钮与控制面板。
● 💾 高效状态管理: 使用 位掩码 (Bitmask) (001, 010, 100) 在一个整型数组中管理同层楼的多种请求（上行、下行、内呼），极大优化了状态判断逻辑。
● 🔄 业务意图分离: 引入“业务方向”与“物理方向”分离机制，完美处理“顺路接客”与“空车赶路”的复杂场景。

📦 安装与运行
由于项目使用了 ES6 Modules 和 Web Workers，浏览器出于安全策略（CORS）限制，无法直接双击 .html 文件打开，必须通过本地服务器运行。
方法 1: 使用 Python (推荐)
如果您安装了 Python，只需在项目根目录下运行：
Bash
# Python 3
python -m http.server

# Python 2
python -m SimpleHTTPServer
然后打开浏览器访问 http://localhost:8000。

2. 右键点击 index.html。
3. 选择 "Open with Live Server"。
🧩 算法原理
1. 任务分配 (主线程)
当用户点击外部按钮时，主线程会遍历所有电梯，计算“代价 (Cost)”：
● 空闲电梯: 代价 = |电梯所在层 - 目标层|
● 同向且顺路: 代价 = |电梯所在层 - 目标层|
● 反向或已错过: 代价 = Infinity (不可用) 或 极高惩罚值
2. 运行决策 (Worker 线程)
电梯内部维护一个长度为 10 的数组 requests，利用位运算存储状态：
● REQ_UP (1): 外部向上请求
● REQ_DOWN (2): 外部向下请求
● REQ_INSIDE (4): 内部下车请求
每次移动前，电梯根据 LOOK 算法 决策：
1. 沿当前方向继续移动，直到前方无任务。
2. 若前方无任务，但反向有任务，则通过“扫描”机制换向。
3. 利用 serviceDirection (业务方向) 锁定空闲电梯的接客意图，避免“空车赶路”时被错误的顺路请求打断。

💡 关于您 floors/README 中的疑问解答
在您上传的 floors/README 文件中，我注意到了您记录的两个关于后端/数据库的问题，这里顺便为您解答，以便您在未来扩展项目时参考：
问题 1：业务对象创建后，一定要手动执行 SQL 吗？
回答：不一定。 在现代后端开发中（如 Java Spring Boot + JPA/Hibernate，或 Python Django/SQLAlchemy），通常使用 ORM (对象关系映射) 技术。
● 自动建表：你只需要定义好代码中的类（Entity），配置好 ddl-auto 为 update，程序启动时会自动在数据库创建或更新表结构，不需要手动写 CREATE TABLE 语句。
● 数据库迁移 (Migration)：生产环境推荐使用 Flyway 或 Liquibase 等工具，它们可以管理 SQL 版本，自动执行变更。
问题 2：特征组的作用，也要手动导入数据库吗？
回答：这取决于“特征组”是代码逻辑还是配置数据。
● 如果是配置数据（比如“电梯A属于高层服务组”），通常建议做成“字典表”或“配置表”。可以通过编写一个 SQL 初始化脚本（如 data.sql），在项目部署时自动执行一次插入操作。
● 如果是动态数据，应该提供一个后台管理界面（Admin UI），让管理员在网页上配置，而不是去改数据库。
